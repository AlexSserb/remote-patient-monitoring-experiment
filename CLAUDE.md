# Remote patient monitoring experiment

В данном проекте будут проводиться эксперименты для выбора нужного алгоритма для системы удаленного мониторинга состояния пациентов. В системе будут 3 роли:
- Врач
- Пациент
- Опекун

Пациенты привязаны к опекунам и к врачам (многие ко многим). Пациенты и опекуны могут заполнять поля дневника здоровья согласно диагнозам конкретного пациента (диагнозы определяют поля). Для более регулярного заполнения будет использоваться система умных уведомлений, которая будет адаптироваться под пациентов и опекунов. Эксперименты проводятся только на пациентах и опекунах.

Алгоритмы, которые будут сравниваться:
- Статический (настраивается для конкретного пользователя согласно его пожеланиям в начале эксперимента)
- Эмпирический (может менять время отправки согласно установленным изначально правилам)
- D-UCB (реализация multi-armed bandit, где автоматы это временные диапазоны)

**D-UCB детали:** 14 автоматов - временные слоты 08:00–22:00 по 1 часу (уведомление отправляется в начале слота). Параметр дисконтирования γ=0.97 (период полураспада ≈23 дня - алгоритм адаптируется к изменениям поведения в течение нескольких недель).

**Эмпирический алгоритм — правила:**
1. Стартует с субъективно лучшего времени пользователя.
2. Скользящее окно 7 дней: если средняя награда < 0.3 → сдвиг времени на +1 слот.
3. Если средняя награда ≥ 0.5 → время зафиксировано (алгоритм доволен результатом).
4. Направление сдвига определяется методом hill-climbing: сначала +1 слот, если через 7 дней хуже - меняет направление на −1.
5. Каждые 30 дней - принудительная переоценка: выбирается слот с наилучшей наградой за последние 30 дней.

Разбиение пользователей:
- 1000 пациентов
- 200 опекунов

Функция награды для многорукого бандита:
```
R = 0.2 * O + 0.3 * D + 0.5 * F
```
- O - открыл ли пользователь систему (0 либо 1)
- D - заполнил ли пользователь дневник (0 либо 1); D=1 только если заполнены все обязательные поля (форма не отправляется без них)
- F - доля заполненных полей дневника; F=0 если D=0 (дневник не открыт или обязательные поля не заполнены)

Длительность эксперимента 180 дней. Один день - это одно уведомление и одна реакция пользователя. После реакции пользователя собираются метрики:
- Функция награды, записанная ранее
- OSR - доля взаимодействия с системой (кол-во взаимодействий с системой делится на количество отправленных уведомлений)
- DCR - доля заполнений дневника (кол-во заполнений дневника делится на кол-во отправленных уведомлений)
- AFFR - средняя доля заполненных полей дневника (сумма количеств заполненных полей дневника делится на сумму всех возможных заполнений полей дневника)
- ART - среднее время реакции пользователя на уведомление (случаи игнорирования уведомления исключаются из расчёта)

Для каждого алгоритма будет выполняться по 10 независимых прогонов с симуляцией 180 дней на 1000 пациентах и 200 опекунах. Из автоматически собранных метрик будут вычисляться средние значения для каждого алгоритма и сравниваться их качество.

Разбиение пациентов по приверженности:
- 20% высокая приверженность
- 50% средняя приверженность
- 30% низкая приверженность

Разбиение опекунов по приверженности:
- 30% высокая приверженность
- 50% средняя приверженность
- 20% низкая приверженность

Опекун заполняет дневник за пациента. Метрики считаются накопительно за все 180 дней эксперимента.

Профили пользователей генерируются через seed для воспроизводимости экспериментов. Каждый прогон использует одни и те же профили, меняется только seed инициализации алгоритмов.

Параметры профиля пользователя для симуляции:
- Субъективное наилучшее время - 7 точек (по одной на каждый день недели), задаётся пользователем при старте.
- Объективное наилучшее время - 7 точек (по одной на каждый день недели), истинный поведенческий оптимум.
- Скорость реакции - логнормальное распределение; пик скорости (минимальная задержка) приходится на объективно лучшее время, вероятность просмотра уведомления тоже максимальна в этот момент.
- Влияние дня недели - 7 мультипликативных коэффициентов, применяются к базовой вероятности реакции.
- Приверженность - определяет базовую вероятность игнорирования уведомления.
- Диагнозы - определяют поля дневника (обязательные и необязательные); описаны в `data/diagnoses.py`.

Симуляцию одного пользователя можно реализовать при помощи генератора, который будет определён с конкретным профилем, а далее будет вызываться некоторое количество раз для проверки конкретной модели. На вход в генератор будет идти время уведомления и день недели. На выходе будут метрики, которые показывают качество работы разработанной модели. Метрики будут включать бинарную величину о заполнении дневника (да/нет), а также долю заполненных полей. Количество полей в дневнике состояния определяется диагнозом пользователя.
